Our Design Overview

Architectural Framework

Employs a complex **three-layer architecture** to separate data storage from on-chain operations, aiming to reduce storage load on nodes.

External Storage Layer
↓
Blockchain Layer (Pointers)
↓
Consensus Layer

Design Philosophy

Adopts a **unified on-chain architecture**. All data and operations are stored directly on the blockchain for simplicity and complete auditability.

Blockchain
(Data + Operations + Consensus)

Node Roles

Defines two distinct node types:

  • Miner Nodes: Authoritative nodes that validate transactions and create blocks.
  • Query Nodes: Non-consensus nodes that handle DNS lookups to improve performance.

Node Architecture

Also utilizes two roles, but their interaction is governed by a robust protocol:

  • Leader Nodes: Authoritative nodes that create blocks in a deterministic, rotating schedule.
  • Normal Nodes: Handle queries and submit transactions to the network.

Storage & Synchronization

Uses direct file system storage with in-memory indexes for fast lookups. A **Gossip protocol** is used to synchronize blocks between nodes.

Technology Stack

The entire system will be implemented in the **Java ecosystem**, leveraging mature libraries for networking, cryptography (Bouncy Castle), and data handling (Jackson).

Acknowledged Implementation Gap

The authors explicitly state their system has **low throughput** and needs a **"more efficient consensus mechanism."**

The paper proposes a framework but does not deliver a defined, fault-tolerant protocol for how nodes should agree on the state of the chain.

Our Cornerstone Solution

Our design's cornerstone is the **Round-Robin Proof of Authority (PoA) with a Timeout**.

This custom protocol provides a deterministic, fork-free, and highly fault-tolerant method for consensus, directly solving the gap left by previous work.
